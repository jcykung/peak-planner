## Goal
Save per-trail planner settings to localStorage so each hike remembers its own date, departure time, linger, pace, and "dude" position in the elevation profile — surviving page reloads and switching between hikes.

## Approach (low-risk: keep globals as a cache of the active hike)
Currently `planDate`, `planDeparture`, `planLinger`, `planPace`, and `activePlayheadPercent` live as flat globals in `peakplanner_settings` and are shared across all hikes. Switching hikes resets the dude to 0 and overwrites your date/time. ~10 functions (`recalculatePlanner`, `fetchWeatherForPlan`, `updateHikerPosition`, `calculateTimelineSplits`, `copySingleItinerary`, …) read these globals.

Rather than thread per-hike state through all of them, I'll introduce a **new per-hike store** and keep the existing globals as an in-memory cache of the *active* hike's values — syncing at the boundaries only.

## Storage change
New key `peakplanner_plans` (kept separate from `peakplanner_wishlist` so a hike can be planned without being wishlisted, and all 20 hikes are supported):
```
hikeId -> { planDate, planDeparture, planLinger, planPace, activePlayheadPercent }
```

## Edits to `index.html`

**1. State + helpers (near lines 2599 / 2677):**
- Add `let plans = {};`
- Add `loadPlansFromLocal()`, `savePlans()`, `defaultPlan()` (returns today's date, `8.0`, `45`, `1.0`, `0.0`), `getPlanForHike(id)` (returns stored plan or default), `applyPlanToGlobals(id)` (copies plan fields into the existing globals), `savePlanForActiveHike()` (writes current globals back to the store).

**2. Boot — `DOMContentLoaded` (~line 2634):** call `loadPlansFromLocal()`; once `activeHike` is known, call `applyPlanToGlobals(activeHike.id)` before `renderActiveHike()` so a reload shows the right values.

**3. `selectHike` (line 3585):** replace `activePlayheadPercent = 0.0;` with `applyPlanToGlobals(hikeId);` — switching to a hike now restores *its* date/time/pace/dude instead of wiping them.

**4. The four plan handlers (lines 3754–3812) + `initPlayheadDrag` endDrag (line 4232):** swap the `saveSettings()` call for `savePlanForActiveHike()` so edits persist per hike.

**5. `loadSettingsFromLocal` / `saveSettings`:** remove the five plan fields from both (now stored per-hike). Settings retains `maxDriveTime`, `activeSort`, `activeDifficultyFilter`, `windyOverlay`, `activeHikeId`, `activeTab`. Old persisted values are silently ignored on load (defaults take over), which is fine since a fresh plan is computed per hike.

**6. Backup export/import (lines 4314–4355):** export now bundles `{ version: 2, wishlist, plans }`; import detects the new wrapped shape vs the legacy plain-wishlist object and restores both. Backward compatible with existing backup files.

## What stays untouched
`renderActiveHike` input-restore block (5677–5685), `updateHikerPosition`, `recalculatePlanner`, weather fetching, SVG — all read the globals, which now reflect the active hike, so no changes needed there.

## Net effect
Each of the 20 hikes independently remembers its planned date, departure time, linger, pace, and dude position. Switching between hikes shows each one's saved state. Reloads and exported backups preserve everything.
